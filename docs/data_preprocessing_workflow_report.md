# ICAS Thermal Data Preprocessing Workflow Report

## Overview

This report documents the post-construction preprocessing workflow used after
`datasets/full_data/` has been built. It describes how the harmonized dataset is
converted into model-ready inputs for ICAS sample-level prediction, including:

1. face and facial ROI annotation,
2. manual quality control and sample exclusion,
3. temperature feature extraction,
4. patient-level train/validation/test splitting, and
5. the final effective sample counts used by downstream models.

In this repository, inference is performed on a **single thermal image at a
time**, so the modeling unit is the **sample**. However, metadata integration
and split assignment are still tracked by patient ID to preserve provenance and
avoid cross-split leakage.

## Starting Point: Harmonized Dataset Bundle

The preprocessing steps below start from the harmonized bundle created by:


## Step 1: Face And ROI Annotation

After `full_data` is built, each image is processed by a two-stage YOLO-based
annotation pipeline:

- a face segmentation model (`4-segmentation`) for whole-face localization,
- a facial ROI model (`8-re_analyze`) for sub-regions such as the eyes, cheeks,
  nose, and forehead.

Run:

```bash
python scripts/face_roi_annotation.py \
  --manifest datasets/full_data/manifest.csv \
  --face-seg-model models/4-segmentation.pt \
  --roi-seg-model models/8-re_analyze.pt \
  --output outputs/annotations
```

Useful optional arguments:

```bash
--device cpu            # or cuda
--face-conf 0.25
--roi-conf 0.25
--padding 20
--roi-size 512
--limit 100             # test run
```

This step produces:

- `outputs/annotations/annotations.json`
- `outputs/annotations/annotations.jsonl`
- `outputs/annotations/summary.csv`
- `outputs/annotations/failures.csv`
- `outputs/annotations/masks/`
- `outputs/annotations/crops/`
- `outputs/annotations/overlays/`
- `outputs/annotations/qc/index.html`

## Step 2: Manual Quality Control And Annotation-Based Exclusion

The repository includes an HTML quality-control interface generated by the
annotation step:

- `outputs/annotations/qc/index.html`

This page is used for manual review of:

- failed face detections,
- anatomically implausible ROI polygons,
- obvious overlay mistakes,
- poor-quality masks/crops that should not be used for modeling.

Samples judged unsuitable during manual review are recorded in:

- `configs/excluded_samples.json`

This file acts as the manual exclusion list consumed by both feature extraction
and CNN training.

## Step 3: Temperature Feature Extraction

Hand-crafted thermal features are extracted from the annotated face and facial
sub-regions using the ROI polygons in `annotations.json`.

Run:

```bash
python scripts/extract_temperature_features.py
```

The explicit form is:

```bash
python scripts/extract_temperature_features.py \
  --annotations outputs/annotations/annotations.json \
  --excluded configs/excluded_samples.json \
  --output datasets/temperature_features.csv \
  --failures datasets/temperature_features_failures.csv \
  --repo-root .
```

This script:

1. loads each temperature CSV matrix,
2. projects face and ROI polygons from image space into temperature-matrix
   space,
3. computes per-region statistics,
4. computes bilateral asymmetry and whole-face thermal descriptors, and
5. writes one row per retained sample.

## Step 4: Patient-Level Train/Validation/Test Split

Although the prediction task is sample-level, the split is generated at the
patient level to prevent images from the same patient appearing in multiple
partitions.

Run:

```bash
python scripts/make_data_split.py
```

The explicit form is:

```bash
python scripts/make_data_split.py \
  --clinical datasets/full_data/patient_clinical_data.csv \
  --features datasets/temperature_features.csv \
  --output configs/data_split.json
```

This script:

1. keeps only patients with valid binary labels,
2. performs a stratified patient-level split with a fixed seed,
3. preserves the ICAS positive rate across train/validation/test, and
4. writes the patient ID lists to `configs/data_split.json`.

## Step 5: Final CNN Training Inputs

The CNN pipeline (`scripts/train_cnn_v2.py`) does not consume every row in
`full_data` directly. Instead, it applies all of the following filters:

1. the patient must belong to the split-specific patient ID list,
2. the patient must have a valid binary label,
3. the sample must not appear in `configs/excluded_samples.json`,
4. the sample must exist in `outputs/annotations/annotations.json`, and
5. the annotation status must be `ok`.

As a result, the final effective cohort used by the current CNN training logic
is:

- **1,371 labeled, annotation-valid, non-excluded samples**
- split into **1,096 train / 140 validation / 135 test**

After any update to the raw dataset matching logic or to `datasets/full_data/`,


## Interpretation For Manuscript Writing

For a paper or thesis, the preprocessing section should distinguish clearly
between:

1. **the harmonized storage dataset**
   - 1,424 samples from 538 patients

2. **the annotation-valid and manually screened sample pool**
   - 1,407 samples after automated annotation and manual exclusion

3. **the final effective labeled CNN cohort**
   - 1,371 samples from 522 labeled patients
   - split into 1,096 train / 140 validation / 135 test

This distinction is important because different filtering steps operate at
different levels:

- patient linkage during dataset construction,
- per-image detection quality during annotation,
- manual sample review after visual QC,
- and label availability during supervised model training.
